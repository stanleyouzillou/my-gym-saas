# Goal
Ship a thin, end-to-end MVP through all layers with hexagonal boundaries:
- Landing (public)
- Member signup (fake payment) → creates Member
- Sessions list (authenticated member)
- Session register (booking) with capacity→waitlist

Keep domain/application pure. Use in-memory adapters first, then wire Prisma adapters behind ports. Deliver minimal web UI to drive the flow.

# Acceptance-first
Create Gherkin features under `features/`:

1) Landing
- Scenario: Visitor sees CTA and can start signup
  Given I am an anonymous visitor
  When I open the landing page
  Then I see a "Join Now" CTA
  And when I click it I am taken to a fake checkout page

2) Member signup (fake payment → creates Member)
- Scenario: Successful fake checkout creates a Member and returns login link
  Given I am on the fake checkout page
  When I submit valid details and confirm payment
  Then a Member is created in the system
  And I receive a login link and land authenticated in the app

3) Sessions list (authenticated)
- Scenario: Member views upcoming sessions
  Given I am an authenticated member
  And sessions exist for my tenant
  When I navigate to the sessions page
  Then I see upcoming sessions with capacity remaining and coach info

4) Session register (booking with waitlist)
- Scenario: Booking succeeds when capacity available
  Given a session with remaining capacity
  When I click Register
  Then I have a Booking with status BOOKED

- Scenario: Waitlist when session is full
  Given a full session at maxCapacity
  When I click Register
  Then I have a WaitlistEntry and no Booking is created

# Architecture & Contracts
- Domain (pure):
  - `scheduling/domain`: Entities: `ClassSession`, `Booking` (status VO), `WaitlistEntry`; VOs: `Capacity` (already exists).
  - Invariants:
    - Booking only when `bookedCount < capacity`
    - Else produce `WaitlistEntry`
- Application ports (interfaces):
  - `ISessionRepo`: `listUpcoming(tenantId)`, `getById(id)`, `save(session)` (if needed for seeding in-memory)
  - `IBookingRepo`: `create(booking)`, `countBySession(sessionId)`, `findByMemberSession(memberId, sessionId)`
  - `IWaitlistRepo`: `enqueue(entry)`, optional `positionFor(memberId, sessionId)`
  - `IMemberRepository` (already domain): used to identify current member
- Use cases:
  - `ListUpcomingSessionsUseCase(tenantId)` → `ClassSession[]`
  - `RegisterForSessionUseCase(memberId, sessionId)` → `Result<{ booking } | { waitlist }>`
- Presentation (API):
  - Controllers (thin):
    - `GET /api/landing` (static content or via web)
    - `POST /api/signup/fake-checkout` → creates Member, returns auth link/token
    - `GET /api/sessions` (auth required) → list
    - `POST /api/sessions/:id/register` (auth required) → booking/waitlist
  - DTOs: request/response types without domain leakage
  - Guards: simple fake auth guard for MVP; later swap with Clerk
- Infrastructure:
  - In-memory adapters for ports (MVP path)
  - Prisma adapters later (contract tests follow pattern already added)
  - Composition root `compose.ts` per bounded context to wire ports↔adapters

# UI Flow (apps/web)
- Pages/Routes:
  - `/` Landing with CTA "Join Now"
  - `/signup/fake` Simple form (email, name) + "Pay" button → calls `POST /api/signup/fake-checkout`
  - After success: store a fake auth token (localStorage) and redirect `/sessions`
  - `/sessions` Fetch `GET /api/sessions` with token; list with Register buttons
  - Register calls `POST /api/sessions/:id/register` → show status BOOKED or WAITLISTED
- Keep styling minimal but clean; use basic CSS or Tailwind if present.

# Implementation Plan (vertical slice)
1) Features (BDD)
- Add 4 `.feature` files: landing, signup, sessions-list, session-register.
- Add step defs in `apps/api/bdd/steps/` (targeted for MVP; reuse in-memory adapters).

2) Application Ports & Use Cases
- Create new ports in `scheduling/application/ports/` for sessions, bookings, waitlist.
- Implement `ListUpcomingSessionsUseCase`, `RegisterForSessionUseCase` in `scheduling/application/use_cases/` using existing `Capacity` VO and new domain artifacts.

3) In-memory Adapters (MVP)
- `scheduling/application/testing/InMemorySessionRepo.ts`
- `scheduling/application/testing/InMemoryBookingRepo.ts`
- `scheduling/application/testing/InMemoryWaitlistRepo.ts`
- Seed a few sessions for tests and for dev server start.

4) Presentation (API)
- Controllers under `scheduling/presentation/`: `SessionsController` with two endpoints.
- Controllers under `memberships/presentation/`: `SignupController` for fake checkout, minimal FakeAuthGuard.
- DTOs for requests/responses.

5) Composition
- `scheduling/infrastructure/compose.ts` registers in-memory repos for MVP and exports tokens.
- `memberships/infrastructure/compose.ts` for signup + existing member repo (in-memory for MVP).
- Import these composition modules into `AppModule` for the MVP.

6) Web UI (apps/web)
- Simple pages per plan, call API endpoints, keep auth as a fake token header.

7) Tests
- Unit: VOs/Entities for new domain objects and use cases.
- Contract: later for Prisma repos mirroring in-memory.
- E2E: Cucumber scenarios driving the flow (can also add supertest-based e2e minimal tests if needed).

# Definition of Done
- Feature files pass with in-memory adapters.
- UI flow works locally end-to-end using fake auth and in-memory repos.
- Ports defined and wired via composition modules; domain untouched by frameworks.
- Docs updated (README snippet explaining MVP run commands).

# Commands (dev)
- API dev (in-memory): `pnpm --filter ./apps/api run start:dev:mem`
- Web dev: `pnpm --filter ./apps/web run dev`
- Tests: `pnpm --filter ./apps/api run test` and Cucumber as configured.
